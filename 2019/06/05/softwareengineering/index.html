敏捷开发

XP: eXtreme Programming

Main ideas:

  • No much documentation. Focus on working code and tests.

  • Implement features one by one.

  • Release code frequently.
  • Work closely with customers.
  • Communicate a lot with team members.

Iteration: 1-3 weeks

User stories

  • wanted features by customer
  • a path through the system
  • written by customers
  • written on index cards
  • format:
    • title
    • acceptance test
    • priority
    • story point: unit of measure for expressing the overall size of a user story, feature, or other piece of work. The raw value of a story point is unimportant. What matters are the relative values.
    • description
  • ideal time: only thing, no interruption
  • elapsed time: total time
  • velocity: rate of progress, sum of story
  • prioritization

Plan each iteration in an iteration meeting.

  • planning poker:
    • Fibonacci sequence
    • estimate story, re-estimate, until converge
  • programmers: one iteration at a time
  • customers: many iterations

One-day tasks.

    • Knowledge management / Reduced product risk

软件测试

Smoke Test

Ensure that the system still runs after making a change.

A quick set of tests run on the daily build.

  • cover most important functionalities of the software.
  • check whether code catches file or smoke.
  • expose integration problems earlier.

Unit Test

Ensure that a module is not broken after making a change.

Integration Test

Verify software quality by testing two or more dependent software modules as a group.

  • Big-bang Integration Testing: All component are integrated together at once

    • Convenient for small systems.
    • Finding bugs is difficult.
    • Due to large number of interfaces that need to be tested, some interfaces could be missed easily.

  • Incremental Integration Testing: Develop a functional “skeleton” system. Design, code, test, debug a small new piece. Integrate this piece with the skeleton.

    • Errors easier to isolate, find, fix
    • System is always in a (relatively) working state
    • May need to create “stub” versions of some features that have not yet been integrated

System Test

Acceptance Test

Regression Test

Ensure that existing code doesn’t get worse as making other improvements. It means re-running test cases from existing test suites to ensure that software changes do not introduce new faults.

Fault vs. Failure vs. Errors

  • Fault: bug in the program (故障)
  • Error: difference between computed value and true value. (错误)
    • Discrepancy between a computed, observed or measured value or condition and the true, specified, or theoretically correct value or condition.
  • failure: program behaves unexpectedly. (失效)
    • Termination of the ability of an element or an item to perform a function as required.

衡量标准

Technical Metrics:

  • Code Size

    • lines of code (assambly > c > java > python)
    • number of classes, functions, files, etc.
    • function points

  • Code Complexity

    • dependencies / coupling / cohesion
    • depth of nesting
    • cyclomatic complexity = number of branches ( if , while , for ) + 1
    • function points

Coupling

  • Dhama’s metric:

    • number of input / output parameters
    • number of global variables used
    • number of modules called / calling

  • Martin’s metric:

    • Ca: Afferent coupling: the number of classes outside this module that depend on classes inside this module
    • Ce: Efferent coupling: the number of classes inside this module that depend on classes outside this module
    • Instability: I = Ce / (Ca + Ce)

Abstractness = (number of abstract classes in module / number of classes in module)

    • Abstract classes or classes hign in hierachy
    • Large entities, comments

文档注释

Javadoc

  • class and interface
    • @author
    • @version
  • method
    • @param
    • @return
    • @exception
  • instance or static variable
    • no tag
  • The first sentence should summarize the purpose of the element
  • For methods, omit the subject and write in the third person narrative form
  • Use the word “this“ rather than “the” when referring to instances of the current class
  • Do not add parentheses to a method or constructor name unless you want to specify a particular signature
  • Keep comments up to date

Bugs and missing features

  • TODO: describes a feature that should be added
  • FIXME: describes a bug in the method
  • XXX: this needs to be thought about some more

Inconsistency

  • correct code, incorrect comment
  • fault in code, correct comment

开发运营和持续集成

DevOps = Development + Optimization

Continuous Integration is a software development practice where members of a team integrate their work frequently

  • Maintain a code repository version control
  • Automate the build
  • Make your build self testing
  • Everyone commits to mainline every day
  • Every commit should build mainline on an integration machine
  • Keep the build fast
  • Test in a clone of the production environment
  • Make it easy for anyone to get the latest executable
  • Everyone can see what’s happening
  • Automate deployment

Top-down Integration: Start with outer UI layers and work inward

Bottom-up Integration: Start with low level data/logic layers and work outward

“Sandwich” Integration: Connect top level UI with crucial bottom level classes

CI server: An external machine that automatically pulls your latest repo code and fully builds all resources.

  • Travis CI

Continuous Delivery: continuously running a deployment pipeline that tests if this software is in a state to be delivered.

  • Continuous Delivery = Continuous Integration + automated test suite

Continuous Deployment: Every change that passes the automated tests is deployed to production automatically.

  • Continuous Deployment = Continuous Delivery + automatic deployment

  • zero-downtime deployment

    1. Deploy version 1 of your service
    2. Migrate your database to a new version
    3. Deploy version 2 of your service in parallel to the version 1
    4. If version 2 works fine, bring down version 1

  • blue-green deployment

    1. Maintain two copies of your production environment (“blue” and “green”)

    2. Route all traffic to the blue environment by mapping production URLs to it

    3. Deploy and test any changes to the application in the green environment
    4. “Flip the switch“: Map URLs onto green & unmap them from blue.

Security terminology

  • Asset: Something of value which has to be protected. The asset may be the software system itself or data used by that system.
  • Attack: An exploitation of a system’s vulnerability. Generally, this is from outside the system and is a deliberate attempt to cause some damage.
  • Control: A protective measure that reduces a system’s vulnerability. Encryption is an example of a control that reduces a vulnerability of a weak access control system
  • Exposure: Possible loss or harm to a computing system. This can be loss or damage to data, or can be a loss of time and effort if recovery is necessary after a security breach.

  • Threat: Circumstances that have potential to cause loss or harm. You can think of these as a system vulnerability that is subjected to an attack.

  • Vulnerability: A weakness in a computer based system that may be exploited to cause loss or harm.

Security risks in moblie application development

  • weak server side controls
    • encrypt the sensitive data
  • insufficient transport layer protection
    • transfer encrypted data
  • poor authentication and authorization
    • manage security logic in API
  • improper session handling
    • set timeout for session
  • sensitive information disclosure
    • expose only required information to API
